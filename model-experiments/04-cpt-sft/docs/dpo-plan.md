# dpo-plan.md — DPO Preference-Pair Dataset

Companion to `spec.md` (umbrella) and `datagen/spec.md` (SFT task catalog,
which this dataset's `chosen` side draws from). Generated by `gen_dpo.jac`
(see `datagen/workflow.md` §1 module graph), output to
`model-experiments/04-cpt-sft/dataset/<run_tag>/releases/dpo_train.jsonl`, run-tag-scoped same
as the SFT dataset (two independent builds, `fresh` and `post_cptv2`).

Existing precedent: `model-experiments/01-sft-dpo/sft_dpo/jacgen/dpo_conversion.jac` already
builds one axis of this (idiomatic-chosen vs Python-shaped-rejected, for
`conversion` only, 147 pairs). This plan generalizes that pattern across all
5 SFT categories and adds four more preference axes beyond idiomatic-vs-not.

## 1. Why more than one axis

A DPO set trained on a single preference axis (idiomatic vs non-idiomatic)
teaches the model one thing: "prefer graph-native Jac over Python-shaped
Jac." That's necessary but not sufficient — a model can write idiomatic-*looking*
Jac that's still wrong, insecure, untyped, or badly named. Each axis below
targets a distinct, real failure mode this project has already seen or is
structurally likely to see (auth leaks are a named bug type in
`datagen/spec.md` §2; type looseness is a named `debug` bug type; naming and
verbosity are the standing code-quality bar this project already holds
itself to). Decision on inclusion/exclusion and relative weight is mine per
the request; rationale given per axis.

## 2. Axes

### 2.1 Idiomatic Jac vs non-idiomatic Jac (flagship axis — 40% of pairs)

The largest axis, by design — this is the one thing every other SFT/DPO
effort on this project has already validated matters (see
`project-sft-dpo-bakeoff` and the existing `dpo_conversion.jac` results).

- **chosen**: graph-native OSP Jac — walkers/nodes/edges/abilities used where
  the problem is actually graph-shaped, `with entry` blocks, typed `has`
  fields, Jac-native control flow.
- **rejected**: same task solved in Python-shaped Jac — plain functions and
  dicts/lists standing in for what should be nodes/edges, no walkers where a
  traversal is the natural fit, defensive Python idiom (e.g. manual
  dict-based "graph" traversal) instead of Jac's native spatial primitives.
- **Generation**: reuse `dpo_conversion.jac`'s pattern (idiomatic seed =
  chosen, `jac py2jac`-transpiled or LLM-Pythonified variant = rejected,
  both compile-gated) extended across `code_gen` and `conversion` seeds, not
  just `conversion`.

### 2.2 Graph-native vs object-shoehorned (15%)

Narrower sibling of 2.1, split out because it's a *design-decision* failure
mode rather than a *style* failure mode — the rejected side still compiles
and runs correctly, it's just modeled wrong (e.g. representing a clearly
relational entity as a nested dict instead of connected nodes). Worth
separating from 2.1 because a model can pass 2.1's idiom-style preference
while still failing to reach for a walker when a traversal is the obviously
correct tool. Sourced from `node_edge_definition`, `walker_traversal`,
`graph_query_patterns` domains in `datagen/spec.md` §1.

### 2.3 Correct vs subtly-wrong (20%)

- **chosen**: code that passes the `jac run` behavioral gate across all test
  cases.
- **rejected**: code that looks plausible, may even compile, but fails a
  held-out test case — sourced directly from `debug` category's buggy
  variants (`datagen/spec.md` §2) before they're paired with their fix. This
  reuses `debug` generation output rather than requiring separate
  generation — the buggy variant already exists; pairing it against its own
  clean seed as `rejected`/`chosen` is close to free.

### 2.4 Secure vs leaky auth (15%)

- **chosen**: `sv_auth_gating`/`sv_multi_user_sharing` code with correct
  `def:priv` gating and per-user data isolation.
- **rejected**: the `auth_leak` bug-type variant from `datagen/spec.md` §2's
  debug taxonomy — endpoint missing `def:priv`, or missing the per-user
  filter, so it returns another user's data.
- Included deliberately as its own axis (not folded into 2.3) because
  security-relevant preference signal is worth training distinctly — a model
  that's merely "usually correct" isn't the same as a model that's reliably
  non-leaky, and leaky-but-otherwise-correct code is exactly the case 2.3
  alone wouldn't catch (the leaky variant may pass every functional test
  case while still being wrong).

### 2.5 Typed vs loosely-typed (10%)

- **chosen**: code with correct, specific type annotations per
  `jac-types` conventions (generics/unions/optionals used where they apply).
- **rejected**: the same code with types widened to `Any`, omitted where
  required, or subtly mismatched (matches the `type_mismatch` bug type from
  `datagen/spec.md` §2, reused the same way as 2.3).

## 3. Explicitly excluded axes (considered, cut)

- **Concise vs verbose**: cut. Verbosity preference is subjective and
  project-style-dependent (this repo's own `ponytail` convention is
  minimal-code, but that's a workflow preference for *this* codebase, not a
  general Jac-language fact — baking it into DPO training data risks
  teaching the model an opinion rather than a language capability). If
  wanted later, this is a separate, explicitly-scoped axis, not bundled in
  here.
- **Well-named vs poorly-named**: cut for this pass. Naming preference pairs
  are the hardest of all these to gate objectively (no compiler signal, no
  behavioral signal, pure LLM-judged aesthetics) and would need a dedicated
  human-review sample to trust — deferred until the other four axes are
  validated and there's headroom to add a judged-only axis.
- **Proper `.impl.jac` split vs monolithic**: cut. Real preference exists
  (`jac-impl-files` recommends splitting past ~150 lines) but is a
  file-organization threshold, not a per-example code-quality judgment —
  doesn't fit the single-example pairwise-preference shape this dataset
  uses. Better handled as a linting rule than a DPO signal.

## 4. Scale

Target 2,000-3,000 pairs per run-tag (roughly 20-25% of the SFT example
count, consistent with the original `strategy.md` distribution table's DPO
share). Split per §2's weights: idiomatic 40% (~1,000), graph-native 15%
(~375), correct-vs-wrong 20% (~500), secure-vs-leaky 15% (~375), typed 10%
(~250).

## 5. Gating

Every `chosen` example must independently pass the same `jac run` gate as
its SFT counterpart (§7 of `../spec.md`). `rejected` examples do **not**
need to fail compilation — most of these axes (2.1, 2.2, 2.4, 2.5) pair a
compiling-but-wrong-in-a-specific-way rejected side against a correct chosen
side; only 2.3 by definition pairs a behaviorally-failing rejected side.
Every pair is deduplicated against every other pair's chosen+rejected code
via `dedup.jac`, and decontaminated against eval holdouts via `decontam.jac`,
same as SFT.

## 6. Output shape

```json
{
  "id": "string",
  "axis": "idiomatic | graph_native | correctness | auth_security | typing",
  "prompt": "string",
  "chosen": "string (Jac code)",
  "rejected": "string (Jac code)",
  "run_tag": "fresh | post_cptv2",
  "generator": "fable-api",
  "generation_date": "timestamp",
  "compiler_pass_chosen": true,
  "compiler_pass_rejected": "true | false (axis-dependent, see §5)"
}
```

`generator` is `fable-api` for every axis — `gen_dpo.jac` runs entirely on
Fable per `../spec.md` §4.1 (preference-pair correctness is precision-critical,
not token-heavy, so it gets the smart model, not the bulk one).

Compatible with `model-experiments/01-sft-dpo/sft_dpo/jacgen/build_dpo_splits.jac`'s
`{prompt, chosen, rejected}` shape for `mlx-lm-lora` DPO training — that
splitter can be reused directly against this dataset's release file, no
format changes needed there.
